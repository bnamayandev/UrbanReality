NeMoTron Agent Design

Multi-Agent Orchestration

Three specialized agents are orchestrated per building request:

1.

Context Agent

Pulls relevant data from PostGIS for the given coordinates: zoning, nearby traffic, tree coverage, business density.

2.

Impact Reasoning Agent

Reasons over context + building specs to produce structured impact scores and natural language explanations.

3.

Citizen Q&A Agent

Conversational agent with full building context loaded. Adapts tone and focus to detected user persona.

Technology Decisions

Stack chosen for 36-hour buildability

Every choice is made to maximize what you can demo in 36 hours while looking polished enough to win.

Frontend

🗺️
CHOSEN: Mapbox GL JS (free tier — 50K loads/month)

3D Map & Building Rendering

Best combination of free tier, 3D building extrusion via fill-extrusion layer, and Toronto building footprint support. Drop a GeoJSON polygon + height = instant 3D building. Looks stunning out of the box.

Alternatives considered: CesiumJS (more powerful but steeper setup), Google Maps (3D locked behind paid tier), Leaflet (no native 3D)
⚛️
CHOSEN: React + Vite

Frontend Framework

Fast setup, component isolation for Builder vs Public mode, fast HMR during dev. Use mapbox-gl React wrapper (react-map-gl) for easy integration.

📊
OPTIONAL: deck.gl (from Uber/NVIDIA)

Data Visualization Layers

If time allows, add deck.gl for heatmap layers showing traffic volume, tree density, and building density across Toronto. Layered on top of Mapbox. Looks incredible and fits the NVIDIA narrative.

Backend

⚡
CHOSEN: FastAPI (Python)

API Server

Python-native, async, auto-generates docs at /docs. Same language as the AI stack so no context switching. uvicorn for serving.

🗄️
CHOSEN: PostgreSQL + PostGIS

Geospatial Database

PostGIS lets you run spatial queries like "find all traffic intersections within 500m of this coordinate" in one SQL line. Ingest Toronto Open Data shapefiles directly. Use SQLAlchemy for Python ORM.

Quick alternative if DB setup is slow: store datasets as in-memory GeoJSON, query with Shapely. Slower but no DB required.

AI / Inference

🧠
CHOSEN: NVIDIA NeMoTron (local on DGX Spark)

Language Model

Running locally = zero API cost, zero latency waiting on external servers, and it's the hardware NVIDIA gave you — use it as the core pitch. Expose via NVIDIA NIM endpoint (http://localhost:8000), call from FastAPI like any OpenAI-compatible API.

Fallback: NVIDIA Build API endpoints (cloud) if local setup has issues during the hackathon.
🔗
CHOSEN: LangChain or direct API calls

Agent Orchestration

For speed: use structured prompts with direct API calls rather than a full agent framework. Keep it simple — one endpoint per agent task, system prompt + context data injection. LangChain only if you need tool-calling loops.

The Core Pipeline

What happens when you hit GET /building/{id}/impact

1
Cache check — query impacts table for this building_id. If found, return immediately. No AI, no ML, instant.
2
Spatial query — spatial.py loads GeoParquet files and runs a 500m radius query. Returns nearby traffic intersections, TTC stops, street trees, businesses, parks, zoning. Cached by coordinate so the same location is only queried once.
3
XGBoost inference — xgb_models.py runs three models in ~10ms. Produces quantitative scores for environmental (kWh), traffic (daily trips), and economic (construction jobs). These are trained on real Toronto data — not estimates.
4
NeMoTron on DGX Spark — building specs + spatial context sent as a structured prompt. NeMoTron reasons over all 5 dimensions and returns JSON. XGBoost scores override the quantitative fields; NeMoTron provides the narrative and covers infrastructure + housing dimensions XGB doesn't handle.
5
Fallback — if NeMoTron is offline, fallback_impact() generates realistic scores mathematically from floor count and footprint. The demo never crashes. XGBoost still runs — only the narrative is missing.
6
Store + return — result saved to impacts table and returned. Next call for this building is instant (step 1).

Building Image Generation — Rehan's Work

LangGraph pipeline + deterministic Pillow renderer

LangGraph Pipeline (3 nodes)
📥

Extract

NeMoTron parses "40-floor glass skyscraper with retail podium" → structured JSON: {building_type, style, floors, size}. Uses LangChain structured output with Pydantic enforcement.

✅

Validate

Pydantic schema check + business rules (skyscrapers ≥20 floors, curtain wall must be glass, podium floors < total). Auto-corrects minor issues. Retries up to 2x on failure.

🖼️

Render

Deterministic Pillow renderer. Same params = pixel-identical PNG every time. Background locked to #E0E0E0 so the 3D pipeline can reliably edge-detect silhouettes.

How To Run

Getting the backend started from scratch

STEP 1 — Database (already done on Wali's machine)
brew install postgresql@17 postgis
brew services start postgresql@17
psql postgres -c "CREATE DATABASE urbanforge;"
psql urbanforge -c "CREATE EXTENSION IF NOT EXISTS postgis;"
STEP 2 — Install dependencies
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
STEP 3 — Download Toronto data + train ML models
python ml/data_pipeline.py # downloads 121 MB of Toronto Open Data
python ml/train_models.py # trains 3 XGBoost models (~30 seconds)
STEP 4 — Start the server
uvicorn main:app --reload
# → http://localhost:8000/docs
