Toronto has a transparency problem with urban development

Thousands of development applications move through the city each year. Residents, businesses, and planners have no unified, visual, data-driven way to understand what's being built and what it means for them.

Who Is Affected
🏗️

Developers & Architects

No fast way to assess site feasibility, zoning compliance, or neighbourhood impact before committing to a location.

🏘️

Residents

No accessible way to understand how a new development affects their rent, transit, or quality of life.

💼

Businesses & Investors

Foot traffic, property value, and demand signals are scattered across incompatible datasets.

Root Cause

The data exists — Toronto Open Data has traffic volumes, building permits, tree canopy, zoning, and more. The problem is it's raw, disconnected, and requires technical expertise to use. There is no intelligent layer that translates data into human-meaningful insights about what a development actually means for the city.

Solution Overview

UrbanForge — AI-Powered Urban Development Intelligence

An interactive map platform where anyone can visualize proposed buildings in 3D anywhere in Toronto, and get instant AI-generated impact analysis — powered by NVIDIA NeMoTron running locally on DGX Spark.

Builder / Developer Interface
📍

Place Anywhere on the Map

Drop a building anywhere in Toronto, configure footprint, height, floors, type, and material.

🏛️

3D Rendering

Real-time 3D building extrusion rendered on Mapbox GL JS with zoning and neighbourhood context.

🤖

AI Impact Analysis Agent

NeMoTron analyzes the site using Toronto Open Data — traffic load, environmental footprint, economic displacement, infrastructure pressure.

🗺️

Smart Suggestions

Agent suggests nearby infrastructure changes: road widening, park needs, transit load implications.

Public / Citizen Interface
👁️

Explore All Active Developments

See every planned and active building project on an interactive Toronto map — not just your neighbourhood.

📋

Building Detail View

Click any building to see: specs, 3D rendering, timeline, developer info, and full impact report.

💬

Persona-Aware Chatbot

Chatbot adapts to context. Investor asks about ROI signals — gets economic data. Resident asks about displacement — gets housing/transit analysis.

🔔

Neighbourhood Alerts (stretch)

Subscribe to notifications when a new development is submitted near your address.

💡 Core Differentiator: The AI doesn't just display data — it reasons over it. "What happens to pedestrian traffic on King St if a 40-storey residential tower goes up here?" is a question only an agent with access to traffic + transit + zoning data can answer.

NeMoTron Agent Design

Multi-Agent Orchestration

Three specialized agents are orchestrated per building request:

1.

Context Agent

Pulls relevant data from PostGIS for the given coordinates: zoning, nearby traffic, tree coverage, business density.

2.

Impact Reasoning Agent

Reasons over context + building specs to produce structured impact scores and natural language explanations.

3.

Citizen Q&A Agent

Conversational agent with full building context loaded. Adapts tone and focus to detected user persona.

Data Flow for Impact Analysis
Step 1
User places building on map → coordinates + specs sent to API
Step 2
PostGIS spatial query: find all datasets within 500m radius
Step 3
Context packaged into structured prompt for NeMoTron
Step 4
NeMoTron runs locally on DGX Spark → zero latency, no API cost
Step 5
Structured JSON response returned → rendered as impact dashboard
Technology Decisions

Stack chosen for 36-hour buildability

Every choice is made to maximize what you can demo in 36 hours while looking polished enough to win.

Frontend

🗺️
CHOSEN: Mapbox GL JS (free tier — 50K loads/month)

3D Map & Building Rendering

Best combination of free tier, 3D building extrusion via fill-extrusion layer, and Toronto building footprint support. Drop a GeoJSON polygon + height = instant 3D building. Looks stunning out of the box.

Alternatives considered: CesiumJS (more powerful but steeper setup), Google Maps (3D locked behind paid tier), Leaflet (no native 3D)
⚛️
CHOSEN: React + Vite

Frontend Framework

Fast setup, component isolation for Builder vs Public mode, fast HMR during dev. Use mapbox-gl React wrapper (react-map-gl) for easy integration.

📊
OPTIONAL: deck.gl (from Uber/NVIDIA)

Data Visualization Layers

If time allows, add deck.gl for heatmap layers showing traffic volume, tree density, and building density across Toronto. Layered on top of Mapbox. Looks incredible and fits the NVIDIA narrative.

Backend

⚡
CHOSEN: FastAPI (Python)

API Server

Python-native, async, auto-generates docs at /docs. Same language as the AI stack so no context switching. uvicorn for serving.

🗄️
CHOSEN: PostgreSQL + PostGIS

Geospatial Database

PostGIS lets you run spatial queries like "find all traffic intersections within 500m of this coordinate" in one SQL line. Ingest Toronto Open Data shapefiles directly. Use SQLAlchemy for Python ORM.

Quick alternative if DB setup is slow: store datasets as in-memory GeoJSON, query with Shapely. Slower but no DB required.

AI / Inference

🧠
CHOSEN: NVIDIA NeMoTron (local on DGX Spark)

Language Model

Running locally = zero API cost, zero latency waiting on external servers, and it's the hardware NVIDIA gave you — use it as the core pitch. Expose via NVIDIA NIM endpoint (http://localhost:8000), call from FastAPI like any OpenAI-compatible API.

Fallback: NVIDIA Build API endpoints (cloud) if local setup has issues during the hackathon.
🔗
CHOSEN: LangChain or direct API calls

Agent Orchestration

For speed: use structured prompts with direct API calls rather than a full agent framework. Keep it simple — one endpoint per agent task, system prompt + context data injection. LangChain only if you need tool-calling loops.

Search these exact terms at open.toronto.ca. MUST-HAVE datasets are critical to core features. NICE-TO-HAVE add depth to the demo.

Conflict Prevention Rules

📄API contract first. Integration Lead writes a shared doc with all endpoint shapes (URL, request body, response schema) before anyone writes code. Frontend builds against mocks. No ambiguity.
🌿One branch per person. No one pushes to main. Integration Lead merges. This prevents conflicts when everyone is coding at 3am.
💬Blockers announced immediately — not "I'll figure it out." If you're stuck for more than 30 minutes, say something. Integration Lead is there to unblock.
🎭Demo data locked at Hour 28. After that, no new data sources. Only UI polish and bug fixes. The demo must work reliably on the three pre-chosen Toronto locations.
🔒Feature freeze at Hour 30. No new features after Hour 30. Shipping what works > adding what breaks.
